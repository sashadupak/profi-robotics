# Разработка алгоритма движения квадрокоптера по траектории

Построение регулятора для движения квадрокоптера по заранее неизвестной траектории.

## Формулировка задачи

Задача была поставлена следующим образом. Необходимо написать алгоритм для движения квадрокоптера по трассе, ширина которой составляет 8м и на которой нанесена линия толщиной 5см. Высота может быть произвольно заданной (до 20м). На полет дается 5 минут, за которые необходимо собрать наибольшее количество виртуальных монеток, расположенных вдоль трассы. Монетки имеют неизвестный радиус, расстояние между монетками также не регламентировано.

<img src="https://github.com/sashadupak/profi-robotics/blob/main/Images/img1.jpg" width="30%"/>

*Трасса в симуляторе Gazebo: вид сверху, трек с нанесённой линией*

Таким образом можно обобщить критерии алгоритма как **точность** следования траектории (нужно обеспечить минимальное отклонение от нанесенной линии) и **скорость**, поскольку нужно за ограниченное время собрать максимальное количество монеток.

**Важные аспекты:**
- Инвариантность алгоритма к высоте
- Стабильность на разных трассах

**Критерии:**
- Точность
- Скорость

## Анализ методов и подходов

**Рассматриваемые методы:**

Рассмотрим наиболее распространенные подходы для решения этой задачи. Прежде всего сердцевина алгоритма — это регулятор, который сможет обеспечить качественное слежение за траекторией. Несмотря на, казалось бы, очевидную задачу существует огромное количество различных подходов для ее решения.

<img src="https://github.com/sashadupak/profi-robotics/blob/main/Images/img2.jpg" width="30%"/>

*Структурная схема системы управления с обратной связью*

Прежде всего это всем известный **ПИД-регулятор**, для линейных систем наиболее распространен **LQR-регулятор** который также может быть применен к квадрокоптеру. Также возможно построение регулятора на основе **метода функции Ляпунова**. Еще существуют адаптивные регуляторы такие как **MPC**, а в последнее время вообще стало актуально синтезировать регулятор для системы рассматривая ее как черную коробку — анализируя зависимость выхода системы от ее входа можно построить регрессионную модель и на ее основании синтезировать регулятор, не имея знаний о физических процессах внутри системы.

## PID-регулятор

Начнем с ПИД-регулятора. Основной плюс — это **простота реализации**, но обратной стороной является **ручная подстройка коэффициентов**. Есть некоторые методы для их расчета: с помощью полинома Ньютона или Баттерворта или по методу Зиглера-Николаса, но на практике подбор коэффициентов работает лучше, что в некоторых случаях просто недопустимо.

<img src="https://github.com/sashadupak/profi-robotics/blob/main/Images/img3.jpg" width="50%"/>

*Simulink: схема PID-регулятора*

**Плюсы:**
- Прост в реализации
- Не требует измерения параметров системы

**Минусы:**
- Использует программное дифференцирование
- Необходим подбор коэффициентов

### Формулы

$$u(t) = K_p e(t) + K_i \int_0^t e(\tau)\,d\tau + K_d \frac{de}{dt}$$

Дискретная реализация:
$$U(n) = K_p E(n) + K_p K_{ip} T \sum_{k=0}^{n} E(k) + \frac{K_p K_{dp}}{T}(E(n) - E(n-1))$$

## LQR-регулятор

Перейдем к такому регулятору как **LQR — линейно квадратичный регулятор**. Он основан на принципе минимизации критерия качества, который определяется матрицами **Q** (штрафы), **R** (веса), **N** — накладывающих ограничения на соответствующие координаты системы.

<img src="https://github.com/sashadupak/profi-robotics/blob/main/Images/img4.jpg" width="30%"/>

*Блок-схема LQR*

**Плюсы:**
- Оптимальное управление
- Возможность выбора критериев регулирования

**Минусы:**
- Только для линейных систем
- Требует точной идентификации системы

### Формулы

Система:
$$\dot{x} = Ax + Bu$$

Функционал качества:
$$J = x^T(t_1)F(t_1)x(t_1) + \int_{t_0}^{t_1}(x^T Q x + u^T R u + 2x^T N u)\,dt$$

Оптимальный закон управления:
$$u = -Kx, \qquad K = R^{-1}(B^T P(t) + N^T)$$

Результатом расчета является матрица **K** оптимальных обратных связей по переменным состояния. Стоит отметить, что этот регулятор работает только для линейных или линеаризованных систем, и предполагает полную идентификацию модели, что в нашем случае сделать проблематично.

## Метод функций Ляпунова

Один из подходов, который не требует полной идентификации модели — это синтез регулятора с использованием **метода функций Ляпунова**. В отличии от других регуляторов он позволяет исключить увеличение ошибки регулирования (т.е. отдаление от желаемой цели из-за перерегулирования).

Его принцип: выбирается функция Ляпунова такая, что ее производная будет отрицательна, и тогда векторы градиента будут противонаправлены векторам поля дифференциального уравнения — таким образом он обеспечивает стремление траекторий к точке равновесия системы.

Минусом является то, что вблизи точки равновесия скорость получается очень незначительной и даже недостаточной чтобы преодолеть внешние силы трения системы — из-за чего образуется **статическая ошибка регулирования**.

<img src="https://github.com/sashadupak/profi-robotics/blob/main/Images/img5.jpg" width="25%"/>

*3D-визуализация функции Ляпунова*

**Плюсы:**
- Функция исключает увеличение ошибки
- Не требуется полная модель системы

**Минусы:**
- Недостаточная величина управления при малых отклонениях

### Формулы

$$V(x_1, x_2) > 0, \quad \dot{V}(x_1, x_2) < 0$$

$$\text{grad}\,V \cdot \dot{x} < 0 \implies \lim_{t \to \infty} x(t) = 0$$

## Model Predictive Control

Совершенно другой подход предполагает **MPC**. Он позволяет реализовать адаптивное регулирование на основе оптимизации модели системы на каждой итерации алгоритма. Позволяет добиться хороших результатов на высоких скоростях. Минусом является более сложная реализация как программная, так и аппаратная, поскольку он требует большие вычислительные мощности.

<img src="https://github.com/sashadupak/profi-robotics/blob/main/Images/img6.jpg" width="50%"/>

*Структура MPC*

**Плюсы:**
- Адаптивное управление
- Оптимизация модели в real-time

**Минусы:**
- Требуются высокие вычислительные мощности
- Требуется идентификация модели

## Теория | Робастное управление

Перейдем к моему решению. Я решил воспользоваться **робастным регулятором**, который, во-первых, позволяет избавиться от программного дифференцирования, во-вторых, предоставляет возможности для подстройки параметров, которые зависят от итерационного шага программы. А также позволяет добиться качественного регулирования по независимой переменной ξ (кси).

<img src="https://github.com/sashadupak/profi-robotics/blob/main/Images/img7.jpg" width="50%"/>

*Simulink: схема робастного регулятора*

### Формулы

$$u_i = \mu_i(\dot{\xi}_i + 2\xi_i)$$
$$\dot{\xi}_i = \sigma_i(-\xi_i + e_i)$$

где $e_i = \{z^* - z,\; \theta^* - \theta,\; \psi^* - \psi,\; \phi^* - \phi,\; \bar{x},\; \bar{y}\}$ — сигналы ошибок, $\mu_i, \sigma_i$ — настраиваемые коэффициенты.

$$\theta^* = \frac{U_5}{U_1}, \quad \psi^* = -\frac{U_6}{U_1}, \quad |\theta_|, |\psi_| \leq \frac{\pi}{6}$$

## Сравнение: PID vs Робастный регулятор

Я хотел бы продемонстрировать сравнение этого регулятора с обычным ПИД-регулятором. Стоит обратить внимание на то, что **ПИД-регулятор** имеет как бы заносы на поворотах, но очень хорошо придерживается референсной точки на прямых. В то время как **робастный регулятор**, наоборот, на прямых имеет большее отставание, зато на поворотах имеет такую способность как бы срезать углы.

То есть при должном подборе коэффициентов он по-хорошему должен проходить повороты по внутреннему радиусу, в то время как ПИД-регулятор такого не умеет и в любом случае в меньшей или большей степени отклоняется по внешнему радиусу поворота. Я не подбирал параметры чтобы регуляторы следовали идеально, иначе было бы не видно их различия в принципе работы.

Я не хочу утверждать, что один регулятор однозначно лучше, а другой хуже, а лишь хочу показать **еще один подход** к решению этой задачи.

## Теория | Проекционное отображение

Еще один значительный момент, о котором я хотел бы рассказать — это **проекционное отображение видеопотока** с камеры с использованием датчиков инерциального измерения. В частности, это гироскоп который измеряет угловую скорость по трем осям, акселерометр который измеряет линейное ускорение по трем осям и иногда компас или магнетометр.

Преобразование этих данных и дальнейшая фильтрация позволяет оценить углы наклона дрона, которые и используются на данном этапе для поворота системы координат камеры. Также осуществляется линейный сдвиг изображения для того, чтобы центр дрона совпадал с центром кадра. Как раз из-за этого сдвига можно заметить образование черных полос по краям в случаях, когда камера наклонена. Для перехода к новому базису используется так называемая **матрица камеры** M, которая содержит фокусные расстояния и их сдвиг относительно центра.

*Калибровочная шахматная доска, демонстрация проекционного отображения c камеры дрона DJI Tello*

### Формулы

$$R_x = \begin{bmatrix}1 & 0 & 0 \\ 0 & \cos\alpha & -\sin\alpha \\ 0 & \sin\alpha & \cos\alpha\end{bmatrix}, \quad R_y = \begin{bmatrix}\cos\beta & 0 & \sin\beta \\ 0 & 1 & 0 \\ -\sin\beta & 0 & \cos\beta\end{bmatrix}, \quad R = R_x R_y$$

$$M = \begin{bmatrix}f_x & 0 & c_x & 0 \\ 0 & f_y & c_y & 0 \\ 0 & 0 & 1 & 0\end{bmatrix}, \quad M^+ = \begin{bmatrix}1/f_x & 0 & -c_x/f_x \\ 0 & 1/f_y & -c_y/f_y \\ 0 & 0 & 0 \\ 0 & 0 & 1\end{bmatrix}, \quad H = MTM^+$$

## Морфологические преобразования

Более интересным мне показалось рассказать про **определение линии на больших высотах**. Дело в том, что по заданию квадрокоптеру задается высота до 20м, а где-то после 10 линия становится очень тонкой и теряется. Сначала я пробовал определять всю трассу целиком, но лучшим решением было применить **расширение линии в зависимости от высоты** дрона. Как видно из видео — чем больше высота дрона над трассой, тем сильнее происходит расширение линии, за счет чего она не теряется и получается довольно хорошо вычислить ее центр.

```python
kernel = np.ones((5,5), np.uint8)
```

*`iterations = position.z / 3` — чем выше дрон, тем сильнее расширяется линия*

## Преобразование медиальной оси

Для того чтобы найти центр линии я использовал **преобразование медиальной оси**, которое позволяет выделить области, равноудаленные от краев, и затем разбить их на точки, по которым и будет следовать дрон. Данный алгоритм иногда выдает небольшие выбросы от центра к краям из-за помех, поэтому используется предварительное размытие изображения.

### Код

```python
dist = cv2.distanceTransform(mask, distanceType=cv2.DIST_L2, maskSize=3)
kernel = cv2.getStructuringElement(cv2.MORPH_CROSS, (3,3))
skeleton = cv2.morphologyEx(dist, cv2.MORPH_TOPHAT, kernel)
skeleton = np.uint8(skeleton * 255)
ret, skeleton = cv2.threshold(skeleton, 0, 255, 0)
_, contours, h = cv2.findContours(skeleton, cv2.RETR_EXTERNAL, cv2.CHAIN_APPROX_NONE)
```

## Поиск ближайшей точки на траектории

После того как были найдены точки, лежащие на линии, вычисляется **ближайшая** из них. Это делается довольно просто: проходом по циклу и вычислением расстояния до каждой из них.

<img src="https://github.com/sashadupak/profi-robotics/blob/main/Images/img10.jpg" width="12%"/>

*Касательная к дуге траектории*

### Код

```python
def distance(self, p1, p2):
    return sqrt((p1[0] - p2[0])**2 + (p1[1] - p2[1])**2)

def closest_point(self, point, array):
    closest = None
    min_val = float('inf')
    for i in range(len(array)):
        if (self.distance(point, array[i]) < min_val):
            min_val = self.distance(point, array[i])
            closest = array[i]
    return closest
```

## Низкочастотный фильтр вибраций

Далее после того, как была определена нужная точка, необходимо применить **низкочастотный фильтр** для того, чтобы снизить раскачку дрона. Дело в том, что из-за резкого перемещения точки даже на незначительные расстояния дрон начинает раскачиваться. Фильтрация позволяет избавиться от этого.

Используется обычный фильтр низких частот первого порядка. На слайде продемонстрирован переход от передаточной функции в образе Лапласа к программной реализации для дискретной системы.
